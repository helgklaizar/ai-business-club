# Структура задачи и сабтасков

Цель: держать процесс **простым** и **по шагам**, особенно для рефакторингов/миграций.

## Каталог задачи

Рекомендуемая структура:

```text
.tasks/<slug>/
  1-task.md
  2-plan.md
  3-result.md
  artefacts/
  inventory/
  subtasks/
  attachments/
```

Правила:
- `1-task.md` и `2-plan.md` — минимально обязательны.
- `3-result.md` — заполняй по факту завершения (или по факту важных сабтасков).
- `artefacts/` — короткие one-pager артефакты (AS-IS, варианты, success checks, модели, миграции, data flow).
- `inventory/` — таблицы/списки (можно больше 1 страницы): модели, маршруты, FE-вызовы, точки авторизации.
- `subtasks/` — **поэтапное внедрение** для рефакторингов/миграций.
- `attachments/` — внешние файлы (скрины, дампы, схемы).

## Когда нужны сабтаски

Сабтаски обязательны, если:
- меняется модель данных/идентичности (например, “вынести admin в отдельную сущность”);
- нужна обратная совместимость / feature-flag / dual-write;
- есть риск “сломать доступ” и нужно безопасное внедрение;
- есть несколько независимых этапов, которые можно проверять по отдельности.

## Формат сабтаска

```text
.tasks/<slug>/subtasks/<NN>-<title>/
  1-task.md
  2-plan.md
  3-result.md
  artefacts/
```

Правила:
- каждый сабтаск — **самодостаточный**: deliverable + check + rollback.
- не начинай следующий сабтаск, пока check предыдущего не пройден (или явно не `N/A`).
- “что поменяли в коде” фиксируй в `3-result.md` сабтаска (или в общем `3-result.md`, если так проще).
