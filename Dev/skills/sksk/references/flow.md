# Флоу разработки (подробнее)

Принцип: **всё — маленькими порциями**. Один артефакт = один вопрос. Один документ ≤ 1 страница.

## 0) Старт
1) Создай `.tasks/<slug>/`.
2) Заполни постановку (без дизайна).
3) Создай `2-plan.md` как “индекс”: список артефактов и сабтасков.

Детали/шаблоны: см. `templates/`.

## 1) Снять AS-IS (текущее состояние)
Цель: перестать “угадывать” систему.

Сделай:
- краткий one-pager `artefacts/ASIS.md` (что сейчас, где границы, где авторизация).
- инвентарь в `inventory/`:
  - модели/сущности и связи;
  - API/маршруты и точки авторизации;
  - FE → API вызовы (если фронт доступен).

Шаблоны:
- `templates/artefacts/asis.md`

Правило:
- если AS-IS не готов, **нельзя** выбирать финальную архитектуру (будет фантазия).

## 2) Discovery research → 1–3 кандидата
Цель: сузить пространство решений и найти сильные подходы **по референсам**, а не “по памяти”.

Вход:
- AS-IS one-pager + инвентарь.
- локальный “корпус проектов” (если есть) + текущий репозиторий.

Выход:
- `artefacts/DISCOVERY.md` (1 страница):
  - 1–3 кандидата;
  - как каждый вписывается в текущую систему (какие границы/данные/контексты затрагивает);
  - trade-offs (по 2–3 пункта);
  - что нужно уточнить у владельцев продукта/системы (≤5 вопросов).

Шаблон:
- `templates/artefacts/discovery.md`

Правило:
- если кандидатов получилось > 3 — ты не сузил поиск, вернись и выбери топ-3.

## 3) “Как должно работать” (success checks)
Цель: зафиксировать поведение так, чтобы потом не спорить о реализации.

Выход:
- `artefacts/OUTCOME.md` (1 страница):
  - success checks (5–12);
  - инварианты (≤15);
  - out-of-scope.

Шаблон:
- `templates/artefacts/outcome.md`

Правило:
- success checks должны быть проверяемыми (“как руками/тестом понять, что ок”).

## 4) Артефакты изменения (модели/миграции/data flow)
Цель: подготовить “скелет изменения”, чтобы план и реализация были безопасными.

Создай (если применимо):
- `artefacts/MODEL.md` — сущности и связи (таблица).
- `artefacts/MIGRATIONS.md` — миграции (порядок, обратимость, совместимость).
- `artefacts/DATAFLOW.md` — датафлоу (FE/клиент → API → auth → side effects → DB).

Шаблоны:
- `templates/artefacts/model.md`
- `templates/artefacts/migrations.md`
- `templates/artefacts/dataflow.md`

Правило:
- любые большие таблицы — в `inventory/` или appendix.

## 5) План → сабтаски
Цель: внедрять безопасно: маленькие шаги, проверки, откаты.

- В `2-plan.md` сделай список сабтасков.
- Для каждого сабтаска создай каталог в `subtasks/` по шаблону.
- Каждый сабтаск должен иметь:
  - deliverable,
  - check,
  - rollback,
  - шаги (без перечисления файлов/классов).

Шаблон:
- `templates/subtasks/subtask.md`

## 6) Реализация по сабтаскам
Цель: один сабтаск → проверка → фиксация результата.

Цикл:
1) Выполни сабтаск.
2) Прогони проверки (из check).
3) Запиши `3-result.md` сабтаска: что сделано, как проверить, что не сделано.
4) Прогони `anti-slop` (см. ниже).

## 7) Anti-slop после каждого шага
См. `anti-slop.md`.

Минимум:
- 1 страница;
- без “воды” и повторов;
- решения привязаны к success checks;
- план шагов содержит check+rollback.
